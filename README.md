main主基座使用aubio库

‌Aubio 是一个开源的音频和音乐分析库‌，主要用于从音频信号中检测音乐事件（如鼓点、音高、节奏等），支持 Python、C 等多种语言接口，官网地址为 https://aubio.org

🎵 核心功能与应用
Aubio 提供了多种专业的音频处理算法，能够将声音信号转化为结构化数据，广泛应用于音乐制作、学术研究和教育领域 。‌‌‌

‌事件检测‌：精准识别音频中的起始点（Onset），如鼓声敲击或音符开始的时刻，支持多种检测算法 。‌‌‌
‌音高与节奏分析‌：
‌音高检测‌：采用 YIN、YINFFT 等算法提取音频频率，适用于乐器调音和旋律分析 。
‌节拍跟踪‌：自动计算音乐的 BPM（每分钟节拍数）并跟踪节拍位置，辅助 DJ 软件和节奏编辑 。‌‌‌
‌特征提取‌：支持 MFCC（梅尔频率倒谱系数）计算、FFT 频谱分析及相位声码器，为机器学习和语音识别提供特征数据 。‌‌‌

结构说明：

1. 音频输入与捕获
音频输入的源头是Android系统的媒体投影（MediaProjection） 服务。

MediaProjectionService.kt：这是一个前台服务，负责请求并维持屏幕/音频捕获的权限。

它的核心作用是创建一个系统级的音频捕获会话，用于实时抓取设备上正在播放的音乐音频数据，为后续处理提供原始音频流。

2. 底层音频处理 (JNI/C层)
这是整个系统的性能核心，所有对音频流的复杂数学计算都在这一层完成。项目通过JNI（Java Native Interface）技术，在C层调用 aubio 库进行分析。

核心文件: pitch.c 和预构建的 libaubio.a 静态库。

初始化 (initTempo): Java层的 AubioProcessor 会调用native方法 initTempo。此方法在C层创建一个 aubio_tempo_t 节拍检测对象，并设置其参数，例如窗口大小为2048采样点，步长为512采样点，采样率为16000Hz。

节拍检测 (getTempo): 这是最关键的环节。当Java层将音频数据（FloatArray）传入 getTempo 方法时，C层代码会执行以下操作：

将音频数据送入 aubio 的节拍检测器。
aubio 库分析音频的能量突变和频谱通量，识别出每个节拍（Beat）的准确位置。
函数最终返回一个浮点数，即当前检测到的BPM（每分钟节拍数）值。
3. 上层特征提取与动作决策 (Kotlin/Java层)
这部分由 MainActivity.kt 主导，它接收来自C层的低阶特征（BPM/Beat），并在此基础上计算出更高级的音乐感知能量（Perceptual Energy, PE），最终依据一个状态机来决定动作。

实时BPM显示: MainActivity 会持续获取C层计算出的BPM值，并更新到UI界面上。

感知能量(PE)计算:

除了BPM，系统还会计算一个“能量”值，用于感知音乐的强弱变化。

它维护一个窗口（PE_WINDOW_SIZE）来存储历史能量数据，并计算出一个最终的感知能量值 currentPeFinal。这个值反映了音乐在较长一段时间内的动态起伏。

动作选择状态机:

候选动作库: 系统预设了一系列动作候选，每个候选都包含一个动作代码（如 "F3", "F4"）和速度等级。

动作切换逻辑: MainActivity 内部维护了一个状态机。它根据当前的节拍计数和感知能量（PE） 的变化，来决定何时以及切换到哪个新动作。

触发条件: 当感知能量（PE）超过一个可调节的阈值（SWITCH_THRESHOLD），并且满足其他状态机条件时，系统就会从候选动作库中选择一个新的动作和速度。

4. 动作执行与可视化 (MotionView)
最终决策出的动作会交给 MotionView 组件来执行和呈现。

MotionView.kt：这是一个自定义View，负责在屏幕上绘制并驱动一个“小人”或图形来执行舞蹈动作。

动作映射: 不同的动作代码（如 "F3"）和速度等级（1-7）会被映射为不同的动画周期（currentPeriodMs）。

动画执行: MotionView 使用 ValueAnimator来控制动画的相位（phase），从而驱动图形按照指定的动作和速度进行运动。

总结
整个流程形成一个清晰的处理链条：

音频输入 (MediaProjection) -> JNI/C层 (Aubio节拍检测) -> 特征提取 (BPM + 感知能量PE) -> 状态机 (动作选择) -> UI渲染 (MotionView动画)

这个架构将计算密集型的音频分析（C层）与逻辑密集型的动作决策（Kotlin层）进行了很好的分离，使得系统能够实时响应音乐节奏并驱动视觉反馈。
